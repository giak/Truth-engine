## Root Cause Analysis

**Hypothesis tested**: DuckDuckGo (web-search MCP) has query complexity limitations.

**Probable causes:**
1. **Query parser overload**: >5 keywords → DuckDuckGo parser fails to construct valid query
2. **Index depth**: DDG index shallower on FR dissident sources (syndicats, ONG, économistes hétérodoxes) vs Google
3. **Ranking algorithm**: DDG prioritizes simple queries, complex dialectical queries ranked lower/timeout

**Validation**: Test 1 v8.7 showed 100% DDG failure on dialectical queries, 100% Google success.

## Solution Architecture

### Intelligent Search Engine Selection (v8.7.1)

**Principle**: Route queries to optimal search engine based on **keyword complexity**.

**Implementation**: system.md §2 STEP 3 (lines 552-560)

```yaml
STEP 3: Execute with intelligent search engine selection (v8.7.1)
  - IF keyword_count > 5 (dialectical/contextualized query):
    → Use WebSearch (Google) DIRECTLY (skip MCP DDG)
    → REASON: DDG systematically fails on complex dialectical queries (v8.7 Test 1: 0/4 success)
    → Track: direct_websearch += 1
  - ELSE (simple query ≤5 keywords):
    → Try MCP (DuckDuckGo) first
    → IF MCP returns [] → Fallback WebSearch (Google)
    → Track: mcp_success, fallback_used per query
```

**Routing logic:**

```
Query generated
    ↓
Count keywords (exclude stopwords)
    ↓
IF keyword_count > 5:          IF keyword_count ≤ 5:
  → WebSearch (Google) direct    → Try MCP (DDG)
  → Track: direct_websearch         ↓
                                  IF results=[]:
                                    → Fallback WebSearch (Google)
                                    → Track: fallback_used
                                  ELSE:
                                    → Track: mcp_success
```

## Key Design Decisions

### 1. Threshold: keyword_count > 5

**Rationale:**
- v8.7 Test 1: All failed queries had 7-8 keywords
- All successful queries had 4 keywords
- Threshold 5 = conservative (captures dialectical queries while preserving DDG for simple queries)

**Alternative considered**: keyword_count > 6 (rejected, too permissive)

### 2. Direct WebSearch for Complex Queries (No DDG Attempt)

**Rationale:**
- 100% DDG failure rate on dialectical queries → no point trying
- Direct routing saves latency (no DDG timeout wait)
- Preserves DDG for simple queries (reduce Google API cost)

**Alternative considered**: Always try DDG first (rejected, wastes time on known failures)

### 3. Preserve Hybrid Fallback for Simple Queries

**Rationale:**
- Simple queries (≤5 keywords) work reliably on DDG (100% Test 1)
- DDG preferable when functional (privacy, cost, MCP integration)
- Fallback to Google only if DDG fails (empty results)

## Risks and Mitigations

### Risk 1: Google API Rate Limits

**Probability**: Medium (if high investigation volume)
**Impact**: High (investigations fail mid-search)

**Mitigation**:
- Track Google API usage per session
- IF approaching rate limit → Warn user, pause investigations
- Future: Implement quota-aware routing (fallback to DDG degraded mode)

### Risk 2: False Negatives (keyword_count ≤5 but complex)

**Probability**: Low (most dialectical queries naturally verbose)
**Impact**: Medium (query fails on DDG, fallback saves but adds latency)

**Mitigation**:
- Existing hybrid fallback (STEP 3 ELSE branch) catches these cases
- Monitor fallback_used metric → Adjust threshold if >10% fallback rate

### Risk 3: DDG Performance Regression

**Probability**: Low (DDG stable for simple queries)
**Impact**: Low (fallback to Google automatic)

**Mitigation**:
- §1c MCP HEALTH CHECK (v8.6.1) detects systematic DDG failures
- Canary query runs before investigation → Sets search_engine=WebSearch if DDG down

## Future Enhancements

### v8.8: Query Complexity Score (beyond keyword count)

**Limitation v8.7.1**: keyword_count is proxy for complexity, not perfect measure

**Proposed**: Multi-factor complexity score:
- Keyword count (weight 0.40)
- Stopword ratio (weight 0.20) - high stopwords = simpler query
- Dialectical keywords (weight 0.20) - presence of "critique", "analyse", "déconstruit"
- Named entities (weight 0.20) - specific actors (CGT, CFDT) vs generic ("syndicats")

**Routing**: IF complexity_score > threshold → Google direct

### v8.9: Search Engine Performance Learning

**Idea**: Track success/failure per search engine per query type

**Adaptive routing**:
```python
IF historical_success_rate[DDG][query_type] > 0.80:
  → Try DDG first
ELSE:
  → Google direct
```

**Benefit**: Auto-adapts to search engine evolution (DDG improvements, Google regressions)
